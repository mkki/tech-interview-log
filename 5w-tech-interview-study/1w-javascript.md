# 1주차 - Javascript

## 스코프·실행 컨텍스트

### Q1. 자바스크립트에서 스코프(Scope)란 무엇이며, `var`와 `let`/`const`를 사용할 때 스코프 관점에서 어떤 차이가 발생하는지 설명해주세요.
- 의도: 식별자 유효 범위 + 블록 스코프 이해
- 키워드: 식별자 접근 범위, var(함수 스코프), let/const(블록 스코프), 전역 오염

<details>
<summary>답변</summary>

스코프는 식별자가 유효한 접근 가능 범위입니다. `var`는 **함수 스코프**만 갖습니다. 함수 내부가 아니면 전역에 붙고, for/if 같은 블록을 무시해서 블록 안에서 선언해도 함수 전체에서 보입니다. 그래서 블록 밖에서도 참조되거나 전역이 오염되기 쉽습니다. `let`과 `const`는 **블록 스코프**입니다. `{}` 단위로 유효하고, 블록 밖에서는 접근할 수 없어 예측하기 쉽고 전역 오염을 줄일 수 있습니다. 실무에서는 스코프를 명확히 하기 위해 `let`/`const` 사용을 권장합니다.

</details>

### Q2. 렉시컬 스코프(Lexical Scope)에 대해 설명하고, 자바스크립트에서 상위 스코프가 결정되는 기준이 무엇인지 말씀해주세요.
- 의도: 렉시컬 스코프 이해
- 키워드: 선언 위치 기준, 정적 스코프, 호출 위치와 무관

<details>
<summary>답변</summary>

렉시컬 스코프는 **정적 스코프**로, 상위 스코프가 **코드를 작성한(선언한) 위치**로 결정됩니다. 함수를 어디서 호출했는지는 상관없고, 정의된 위치 기준으로 스코프 체인이 만들어집니다. 따라서 함수 객체가 생성될 때 이미 외부 렉시컬 환경 참조가 고정되고, 런타임에 호출 위치를 바꿔도 그 참조는 변하지 않습니다. 동적 스코프(호출 위치 기준)와 구분되는 자바스크립트의 중요한 특성입니다.

</details>

### Q3. 실행 컨텍스트(Execution Context)의 생성 과정을 '소스코드 평가'와 '실행' 단계로 나누어 상세히 설명해주세요.
- 의도: 실행 컨텍스트 생성 과정 이해
- 키워드: 평가 단계, 실행 단계, 메모리 등록, 콜 스택

<details>
<summary>답변</summary>

실행 컨텍스트는 **소스코드 평가**와 **실행** 두 단계로 만들어집니다. **평가 단계**에서는 실행 컨텍스트를 생성하고, 렉시컬 환경을 만들며 변수·함수 선언을 환경 레코드에 등록합니다. 이때 `var`는 `undefined`로 초기화되고, `let`/`const`는 초기화되지 않은 채로 등록됩니다. **실행 단계**에서는 소스코드가 순차 실행되며, 변수 할당이나 함수 호출이 일어납니다. 실행 컨텍스트는 콜 스택에 push되어 현재 실행 중인 코드를 관리하고, 종료 시 pop됩니다.

</details>

### Q4. 렉시컬 환경(Lexical Environment)의 구성 요소인 '환경 레코드'와 '외부 렉시컬 환경 참조'가 스코프 체인을 형성하는 데 어떤 역할을 하나요?
- 의도: 렉시컬 환경 구조 이해
- 키워드: 환경 레코드, 외부 참조, 스코프 체인

<details>
<summary>답변</summary>

**환경 레코드(Environment Record)**는 현재 스코프에 선언된 식별자와 그 값을 담습니다. 변수나 함수를 찾을 때 먼저 이 레코드를 검색합니다. **외부 렉시컬 환경 참조(Outer Environment Reference)**는 상위 스코프의 렉시컬 환경(Lexical Environment)을 가리킵니다. 현재 환경 레코드에서 식별자를 찾지 못하면 이 참조를 따라 상위 환경을 찾고, 이를 반복해 결국 전역까지 이어집니다. 이 연결이 **스코프 체인**이고, 식별자 검색은 이 체인을 따라 올라가며 이루어집니다.

</details>

### Q5. 호이스팅(Hoisting) 현상이 발생하는 근본적인 원인은 무엇이며, 변수 종류(`var` vs `let`/`const`)에 따라 참조 결과가 다른 이유를 설명해주세요.
- 의도: 호이스팅 본질 이해
- 키워드: 선언 등록, 초기화 차이, TDZ

<details>
<summary>답변</summary>

호이스팅은 **실행 전 소스코드 평가 단계**에서 선언문이 먼저 실행되어 환경 레코드에 식별자가 등록되기 때문에 발생합니다. 선언이 코드 상단으로 “끌어올려진 것처럼” 동작하는 게 아니라, 평가 시점에 이미 등록되기 때문입니다. `var`는 이때 **선언과 동시에 `undefined`로 초기화**되므로 선언부 이전에 참조해도 `undefined`가 나옵니다. `let`/`const`는 **선언만 등록되고 초기화는 실제 선언문이 실행될 때** 이루어지므로, 그 사이 구간에서 참조하면 초기화되지 않은 상태로 접근하게 되어 **TDZ(Temporal Dead Zone)** 에러가 발생합니다.

</details>

## this·바인딩·함수

### Q6. 자바스크립트 `this` 키워드의 특징을 일반 함수 호출과 객체의 메서드 호출 상황을 비교하여 설명해주세요.
- 의도: this 바인딩 규칙 이해
- 키워드: 호출 방식 결정, 전역 객체, 메서드 호출

<details>
<summary>답변</summary>

`this`는 **호출 방식**에 따라 결정됩니다. **일반 함수 호출**에서는 기본적으로 전역 객체(브라우저의 `window`, Node의 `global`)가 바인딩됩니다. strict mode에서는 `undefined`입니다. **메서드 호출**에서는 그 메서드를 **소유한 객체**가 `this`가 됩니다. `obj.fn()`처럼 호출하면 `fn` 내부의 `this`는 `obj`를 가리킵니다. 정리하면, 선언 위치가 아니라 “누가 어떻게 호출했는지”가 `this`를 결정합니다.

</details>

### Q7. 명시적 `this` 바인딩 방법인 `call`, `apply`, `bind`의 차이점과 각각 어떤 상황에서 사용하는 것이 적절한지 말씀해주세요.
- 의도: 명시적 this 바인딩 이해
- 키워드: call/apply 즉시 실행, bind 함수 반환, 인자 전달 방식

<details>
<summary>답변</summary>

세 메서드 모두 첫 번째 인자로 `this`를 지정합니다. **call**은 인자를 쉼표로 나열해 넘기고, **apply**는 인자 배열 하나로 넘깁니다. 둘 다 **즉시 호출**됩니다. 유사 배열에 배열 메서드를 쓰거나, 생성자 상속 패턴에서 `Parent.call(this)`처럼 쓸 때 call/apply를 씁니다. **bind**는 호출하지 않고 **새 함수를 반환**하며, 그 함수의 `this`가 항상 지정한 값으로 고정됩니다. 이벤트 핸들러나 콜백에서 `this`를 고정해 두고 나중에 호출할 때 사용합니다.

</details>

### Q9. `strict mode`를 적용했을 때 일반 함수 내부의 `this` 값은 어떻게 변하며, 이러한 제약이 실무적으로 어떤 실수를 방지해줄 수 있나요?
- 의도: strict mode 이해
- 키워드: this undefined, 전역 바인딩 방지, 에러 강화

<details>
<summary>답변</summary>

strict mode에서는 **일반 함수 호출 시 `this`가 `undefined`**가 됩니다. 비 strict에서는 전역 객체가 바인딩되던 것과 다릅니다. 이렇게 되면 전역 객체를 실수로 수정하는 일을 막을 수 있고, 메서드가 아닌 함수에서 `this`를 기대하고 썼을 때 조기에 에러가 나서 디버깅이 쉬워집니다. 또한 암묵적 전역 변수 생성 금지, `delete` 제한 등으로 에러를 강화해 실수를 줄입니다. 실무에서는 기본적으로 strict mode 사용을 권장합니다.

</details>

### Q10. 함수 선언문과 함수 표현식 중 어떤 방식을 선호하시나요? 호이스팅 관점에서 답변해주세요.
- 의도: 함수 정의 방식 차이
- 키워드: 선언문 호이스팅, 표현식 TDZ, 예측 가능성

<details>
<summary>답변</summary>

**선호하는 쪽은 함수 표현식**입니다. 선언문은 전체가 호이스팅되어 선언 전에 호출해도 동작하지만, 그만큼 코드 순서와 실행 순서가 달라져 예측이 어렵습니다. 표현식은 변수에 할당되는 형태라 변수만 호이스팅되고 함수는 할당 시점에 붙습니다. `const`/`let`이면 TDZ 때문에 선언 전 호출 시 에러가 나서, “선언 위에서 호출” 같은 실수를 막을 수 있습니다. 따라서 위에서 아래로 읽는 흐름과 실행이 맞고, 의도가 분명해 유지보수에 유리합니다.

한편 **호이스팅의 이점**은 “선언 위치보다 먼저 호출 가능”하다는 점입니다. 실무에서는 이걸 활용하는 경우가 많습니다. **전역/모듈 초기화 함수**를 파일 하단에 두고 상단에서 호출할 수 있고, **라이브러리 헬퍼 API**를 사용하는 코드를 선언보다 위에 배치할 수 있으며, **순환 참조**가 생기는 구조에서도 선언문이면 안정적으로 참조할 수 있습니다. **이벤트/콜백 바인딩** 시에도 호이스팅 덕분에 바인딩 순서를 신경 쓰지 않고 설계할 수 있습니다. 즉, 호이스팅 덕분에 코드 구조를 더 유연하게 설계할 수 있어 실무 라이브러리에서 적극 활용됩니다.
</details>

### Q11. bind로 고정된 함수의 this를 다시 바꿀 수 있나요?
- 의도: bind 동작 이해
- 키워드: this 고정, 재바인딩 불가, 새로운 함수

<details>
<summary>답변</summary>

**일반적으로는 바꿀 수 없습니다.** `bind`가 반환하는 것은 원본과 다른 **새로운 함수**이고, 스펙상 이 함수의 `[[BoundThis]]`가 bind 시 넘긴 값으로 고정됩니다. call/apply로 다시 `this`를 넘겨도 이 bound 함수의 `this`는 무시되고, 항상 bind 시 지정한 값이 사용됩니다. **예외는 `new`로 호출할 때**입니다. bound 함수를 `new`로 호출하면 생성자로서 동작하며, `this`는 새로 만든 인스턴스로 바인딩됩니다. 다른 `this`로 쓰고 싶다면 원본 함수를 다시 bind하거나, `new`로 호출하는 방식(생성자 사용 시)을 고려하면 됩니다.

</details>

- 의도: 런타임 이해
- 키워드: JS 엔진, Web API, libuv, 환경별 API

<details>
<summary>답변</summary>

둘 다 **JavaScript 엔진**(V8 등)으로 코드를 실행하는 건 동일합니다. **브라우저**는 DOM, `fetch`, `setTimeout`/`setInterval`, 이벤트 등 **Web API**를 제공하고, 이들은 엔진 밖에서 비동기 작업을 처리한 뒤 태스크 큐로 콜백을 넘깁니다. **Node.js**는 브라우저 API 대신 **libuv** 기반의 이벤트 루프와 **Node 전용 API**(파일 시스템 `fs`, HTTP `http`/`https`, `path`, `crypto` 등)를 제공합니다. 즉, 엔진은 같고 비동기·I/O를 담당하는 API와 런타임 구성 요소가 환경마다 다릅니다.

</details>

## 클로저

### Q12. 클로저란 무엇인가요?
- 의도: 클로저 개념 이해
- 키워드: 외부 변수 참조, 렉시컬 환경 유지

<details>
<summary>답변</summary>

**클로저**는 함수가 자신이 선언된 렉시컬 스코프를 기억해, 그 스코프가 실행 컨텍스트 상으로는 종료된 뒤에도 그 스코프의 변수에 접근할 수 있는 현상입니다. 내부 함수가 외부 함수의 변수를 참조하고, 그 내부 함수가 외부로 반환되거나 다른 곳에 전달되면, 외부 함수의 렉시컬 환경이 가비지 컬렉션되지 않고 유지됩니다. 그래서 “함수 + 그 함수가 참조하는 외부 변수(환경)”가 함께 살아 있는 구조라고 보면 됩니다.

</details>

### Q13. 실행 컨텍스트 종료 후에도 외부 변수 접근 가능한 이유는?
- 의도: 메모리 구조 이해
- 키워드: 힙 저장, 참조 유지, GC 대상 제외

<details>
<summary>답변</summary>

실행 컨텍스트가 종료되어도 **내부 함수가 외부 렉시컬 환경을 참조**하고 있기 때문입니다. 렉시컬 환경은 객체처럼 **힙**에 저장되고, 내부 함수의 `[[Environment]]`가 그 환경을 가리킵니다. 따라서 외부 컨텍스트는 pop되어도, 그 환경을 참조하는 내부 함수가 살아 있는 한 해당 환경은 GC 대상이 되지 않고 유지됩니다. 그래서 클로저를 통해 “종료된 함수”의 변수에 계속 접근할 수 있습니다.

</details>

### Q14. 클로저를 활용한 캡슐화의 장점은?
- 의도: 설계 이해
- 키워드: 데이터 은닉, private 상태, 안정성

<details>
<summary>답변</summary>

클로저로 **외부에 노출하지 않고 상태를 감춘 private 같은 공간**을 만들 수 있습니다. 함수 스코프 안에 변수를 두고, 그 변수를 읽거나 바꾸는 메서드만 반환하면, 그 변수에는 오직 그 메서드를 통해서만 접근할 수 있습니다. 전역 변수나 객체 프로퍼티로 두면 누구나 덮어쓸 수 있는 문제를 피할 수 있고, 초기화·검증 로직을 한 곳에 모아 두어 상태가 항상 일관되게 유지되도록 할 수 있습니다. class의 private 필드가 도입되기 전에도 이 패턴이 널리 쓰였습니다.

</details>

### Q15. 클로저의 단점은?
- 의도: 트레이드오프 이해
- 키워드: 메모리 유지, 성능 저하, 디버깅 어려움

<details>
<summary>답변</summary>

클로저가 **외부 렉시컬 환경을 참조하는 동안** 그 환경은 GC되지 않아 **메모리가 더 오래 유지**됩니다. 클로저를 많이 쓰거나 큰 데이터를 닫아두면 메모리 사용이 늘어날 수 있습니다. 스코프 체인이 길어져 **식별자 탐색 비용**이 조금 늘 수 있고, 닫혀 있는 변수가 어디서 어떻게 쓰이는지 추적하기 어려워 **디버깅**이 난해해질 수 있습니다. 모든 상태를 클로저로만 감싸면 **외부에서 직접 수정이 불가**해지고, **상속·믹스인·상태 동기화** 같은 패턴을 적용하기 어렵습니다. 그래서 필요한 스코프만 참조하도록 설계하고, 사용이 끝난 클로저는 참조를 끊어 주는 것이 좋습니다.

</details>

### Q16. 클로저 메모리 해제 방법은?
- 의도: 메모리 관리 이해
- 키워드: 참조 제거, null 할당, GC

<details>
<summary>답변</summary>

클로저가 참조하는 **외부 환경**은 “그 클로저를 참조하는 것”이 없어지면 GC 대상이 됩니다. 따라서 **클로저를 담고 있던 변수에 `null`이나 다른 값을 할당**해 참조를 끊으면, 그 클로저와 연결된 렉시컬 환경에 대한 참조가 사라져 GC가 수거할 수 있습니다. 이벤트 리스너를 제거하거나, 클로저를 보관하던 객체/배열을 비우는 것도 같은 효과입니다. 사용이 끝난 클로저는 가능한 한 참조를 제거해 두는 것이 좋습니다.

</details>

## 이벤트 루프·비동기

### Q17. JS가 싱글 스레드라는 의미는?
- 의도: 실행 모델 이해
- 키워드: 한 번에 하나 실행, 콜 스택

<details>
<summary>답변</summary>

JavaScript는 **한 번에 하나의 실행 컨텍스트만** 콜 스택에서 실행됩니다. 멀티 스레드처럼 여러 코드가 동시에 실행되는 것이 아니라, 스택에 쌓인 함수가 위에서부터 하나씩 실행되고 pop될 뿐입니다. 그래서 어떤 코드가 오래 돌면 그동안 다른 코드는 실행되지 않고, 블로킹이 발생합니다. 비동기 I/O나 타이머는 엔진 밖(Web API, libuv 등)에서 처리되고, 완료된 콜백만 큐를 통해 스택에 들어와 실행됩니다.

</details>

### Q18. 비동기가 필요한 이유는?
- 의도: 비동기 필요성 이해
- 키워드: UI 블로킹 방지, 사용자 경험

<details>
<summary>답변</summary>

네트워크 요청이나 파일 읽기 같은 I/O는 완료까지 시간이 걸립니다. 이걸 동기로 처리하면 그동안 **콜 스택이 막혀** UI가 멈추고, 사용자 입력에 반응하지 못합니다. 비동기로 처리하면 요청만 걸어 두고 메인 스레드는 다른 일을 하다가, 완료 시 콜백으로 결과만 받아 처리할 수 있어 **UI 블로킹을 막고** 사용자 경험을 유지할 수 있습니다. 싱글 스레드 환경에서 동시성은 이벤트 루프와 큐를 통한 비동기 처리로 구현됩니다.

</details>

### Q19. 이벤트 루프 역할은?
- 의도: 이벤트 루프 이해
- 키워드: 콜 스택 확인, 큐 처리

<details>
<summary>답변</summary>

이벤트 루프는 **콜 스택이 비었을 때** 태스크 큐(마이크로태스크 큐 우선)에서 대기 중인 콜백을 꺼내 **콜 스택에 넣어 실행**하는 역할을 반복합니다. 싱글 스레드이기 때문에 “지금 실행 중인 코드”가 끝나야 다음 작업을 할 수 있는데, 비동기 작업이 끝나면 콜백이 큐에 들어가고, 이벤트 루프가 “스택이 비었나?”를 확인한 뒤 큐의 콜백을 하나씩 실행합니다. 이렇게 해서 비동기 결과를 메인 스레드에서 순차적으로 처리합니다.

</details>

### Q20. setTimeout 동작 과정은?
- 의도: Web API 이해
- 키워드: 타이머 등록, 큐 이동

<details>
<summary>답변</summary>

`setTimeout(callback, delay)`를 호출하면 **콜백과 지연 시간이 타이머(Web API / libuv)에 등록**됩니다. 메인 스레드는 그대로 다음 코드를 실행하고, 별도로 타이머가 동작하다가 delay가 지나면 **콜백이 태스크 큐(매크로태스크 큐)에 넣어집니다.** 이벤트 루프가 콜 스택이 비었을 때 이 큐에서 콜백을 꺼내 실행합니다. 따라서 delay는 “정확히 그만큼 뒤에 실행”이 아니라 “그 시간이 지난 뒤, 스택이 비면 가능한 빠른 시점에 실행”을 의미합니다.

</details>

### Q21. 브라우저는 멀티 스레드라는 의미는?
- 의도: 환경 이해
- 키워드: Web API 별도 처리

<details>
<summary>답변</summary>

JavaScript 실행 자체는 **싱글 스레드**이지만, 브라우저 전체는 **멀티 스레드**로 동작합니다. 타이머, 네트워크, DOM 조작, 파일 접근 같은 **Web API**는 브라우저가 별도 스레드나 프로세스에서 처리합니다. 그래서 `setTimeout`을 호출하면 타이머 관리 스레드가 기다리고, fetch를 하면 네트워크 스레드가 요청을 처리합니다. 완료되면 콜백이 큐에 들어가고, JS 스레드는 이벤트 루프로 그 콜백만 실행합니다. 즉, “JS 코드 실행”은 한 스레드, “그 밖의 작업”은 브라우저가 여러 스레드로 처리한다는 뜻입니다.

</details>

### Q22. 콜 스택이 바쁠 때 setTimeout은?
- 의도: 큐 동작 이해
- 키워드: 대기 상태 유지

<details>
<summary>답변</summary>

콜백은 **delay가 지나면 태스크 큐에 들어가고**, 그 후에는 **콜 스택이 비어 있을 때만** 이벤트 루프에 의해 실행됩니다. 따라서 콜 스택이 바쁘면(긴 동기 코드나 무한 루프가 돌면) 이미 큐에 들어온 setTimeout 콜백도 실행되지 않고 **대기 상태로 남습니다.** 지정한 delay는 “최소 대기 시간”일 뿐, 스택이 비기 전까지는 실행이 지연됩니다.

</details>

### Q23. 출력 순서 문제 설명
- 의도: 이벤트 루프 흐름 이해
- 키워드: 스택, 큐, 실행 순서

<details>
<summary>답변</summary>

동기 코드가 먼저 전부 실행된 뒤, **마이크로태스크 큐**(Promise then/catch, queueMicrotask)가 **매크로태스크 큐**(setTimeout, I/O 콜백)보다 먼저 비워집니다. 그래서 `console.log` → `Promise.then` → `setTimeout` 순으로 나오는 식의 출제가 많습니다. 한 턴에서 스택이 비면 마이크로태스크를 모두 실행한 다음, 그 다음 매크로태스크 하나를 실행하고, 다시 스택이 비면 마이크로태스크 전부 실행하는 식으로 반복됩니다. 이 순서를 이해하면 출력 순서 문제를 맞출 수 있습니다.

</details>

### Q24. 이벤트 루프 실행 주기
- 의도: 내부 동작 이해
- 키워드: tick, 반복 검사

<details>
<summary>답변</summary>

이벤트 루프는 **한 번의 tick**마다 “콜 스택이 비었는지” 확인하고, 비었으면 마이크로태스크 큐를 모두 처리한 뒤, 매크로태스크 큐에서 하나를 꺼내 실행합니다. 이 한 사이클이 끝나면 다시 스택 확인 → 큐 처리를 반복합니다. 주기는 환경·부하에 따라 달라지며, “가능한 한 빨리” 검사하지만 CPU를 계속 점유하지는 않고, 작업이 있을 때만 동작하는 식으로 구현됩니다.

</details>

### Q25. Node.js의 비동기 처리 구성 요소
- 의도: Node 이해
- 키워드: libuv, 이벤트 루프

<details>
<summary>답변</summary>

Node.js는 V8 엔진으로 JS를 실행하고, **libuv**가 **이벤트 루프**와 비동기 I/O(파일, 네트워크, 타이머 등)를 담당합니다. libuv는 OS의 비동기 API(epoll, kqueue, IOCP 등)를 쓰면서 이벤트 루프를 돌리고, 완료된 작업의 콜백을 JS 쪽으로 넘깁니다. 그래서 타이머, fs, net 같은 네이티브 모듈이 비동기로 동작할 수 있습니다. Promise·async/await는 JS 엔진의 마이크로태스크 큐를 사용하고, I/O 완료 이벤트는 libuv를 통해 매크로태스크처럼 들어갑니다.

</details>

### Q26. 콜백 사용하는 이유
- 의도: 비동기 처리 이해
- 키워드: 완료 후 실행

<details>
<summary>답변</summary>

비동기 작업은 “언제 끝날지 모르기 때문에” 결과를 **나중에** 받아야 합니다. 그 “나중에 할 일”을 **콜백 함수**로 넘겨 두면, 작업이 완료된 시점에 런타임(또는 API)이 그 콜백을 호출해 줍니다. 그래서 “완료 후 실행” 로직을 표현할 수 있습니다. 단, 콜백을 중첩하면 가독성이 떨어지므로 Promise·async/await로 단순화하는 것이 일반적입니다.

</details>

## Promise·async/await

### Q27. Promise란?
- 의도: Promise 이해
- 키워드: 상태(pending, fulfilled, rejected)

<details>
<summary>답변</summary>

Promise는 **비동기 작업의 최종 완료(또는 실패)** 를 나타내는 객체입니다. 상태는 **pending**(대기), **fulfilled**(이행), **rejected**(거부) 세 가지이고, 한 번 fulfilled나 rejected가 되면 바뀌지 않습니다. 생성자에 넘긴 executor에서 `resolve`/`reject`를 호출해 상태를 바꾸고, `.then`/`.catch`로 결과나 에러를 처리합니다. 콜백 지옥을 피하고, 연쇄·병렬 처리·에러 전파를 일관되게 다루기 위해 사용합니다.

</details>

### Q28. resolve/reject 역할
- 의도: 상태 변경 이해
- 키워드: 성공, 실패

<details>
<summary>답변</summary>

**resolve(value)** 는 Promise를 **fulfilled**로 바꾸고, `value`를 결과값으로 저장합니다. 이후 `.then`에 전달된 콜백이 그 값과 함께 호출됩니다. **reject(reason)** 은 Promise를 **rejected**로 바꾸고, `reason`을 사유로 저장합니다. 이후 `.catch`나 `.then`의 두 번째 인자로 전달됩니다. 둘 다 **한 번만** 유효하고, 이미 settled된 Promise에 다시 호출해도 상태는 변하지 않습니다.

</details>

### Q29. await 동작
- 의도: async/await 이해
- 키워드: Promise 대기, 흐름 중단

<details>
<summary>답변</summary>

`await`는 **async 함수 안에서만** 쓸 수 있고, 오른쪽의 Promise가 **settled될 때까지** 해당 async 함수의 실행을 일시 중단합니다. Promise가 fulfilled면 그 값이 await 표현식의 결과가 되고, rejected면 예외가 던져져서 try/catch로 잡을 수 있습니다. 중단된 동안 메인 스레드는 다른 스크립트·이벤트를 처리할 수 있어, 동기 코드처럼 보이면서도 논블로킹으로 동작합니다. 내부적으로는 Promise then 체인으로 변환되는 형태입니다.

</details>

### Q30. Promise 체이닝 장점
- 의도: 코드 구조 이해
- 키워드: 가독성, 순차 처리

<details>
<summary>답변</summary>

`.then()`이 새 Promise를 반환하므로 `.then().then()`처럼 **체이닝**할 수 있습니다. 순차 비동기 단계를 “위에서 아래로” 쓰게 되어 **가독성**이 좋아지고, 각 단계에서 반환한 값이 다음 then의 인자로 넘어가 **순차 처리**를 자연스럽게 표현할 수 있습니다. 콜백을 중첩하는 것보다 에러 전파도 `.catch()` 한 번으로 처리할 수 있어 유지보수가 쉽습니다. async/await는 이 체이닝을 동기 코드 스타일로 쓴 문법입니다.

</details>

### Q31. 태스크 vs 마이크로태스크
- 의도: 큐 우선순위 이해
- 키워드: microtask 우선

<details>
<summary>답변</summary>

**매크로태스크**(태스크)는 setTimeout, setInterval, I/O, UI 이벤트 등의 콜백이 들어가는 큐이고, **마이크로태스크**는 Promise의 then/catch/finally, queueMicrotask 등이 들어가는 큐입니다. 이벤트 루프 한 턴에서 **콜 스택이 비면 먼저 마이크로태스크 큐를 전부 비운 뒤**, 그다음 매크로태스크 **하나만** 실행합니다. 그래서 같은 턴 안에서는 마이크로태스크가 항상 매크로태스크보다 먼저 실행됩니다. 출력 순서 문제는 이 우선순위를 알아야 풀 수 있습니다.

</details>

### Q32. async/await 에러 처리
- 의도: 예외 처리 이해
- 키워드: try/catch

<details>
<summary>답변</summary>

async 함수 안에서 `await`한 Promise가 reject되면 **예외가 throw**됩니다. 그래서 동기 코드처럼 **try/catch**로 감싸면 됩니다. try 블록 안의 await에서 에러가 나면 catch 블록으로 넘어가고, catch에서 처리하거나 다시 throw할 수 있습니다. 반환값이 없으면 async 함수는 rejected Promise를 반환하므로, 호출 쪽에서 `.catch()`로 처리할 수도 있습니다. 동기·비동기 에러를 같은 try/catch 패턴으로 다룰 수 있어서 에러 처리가 단순해집니다.

</details>

### Q33. 여러 요청 처리 방법
- 의도: 병렬 처리 이해
- 키워드: Promise.all

<details>
<summary>답변</summary>

여러 비동기 요청을 **동시에** 보내고 모두 완료된 뒤 처리하려면 **Promise.all(iterable)** 을 씁니다. 인자로 넘긴 Promise가 모두 fulfilled면 결과 배열이 담긴 Promise가 이행되고, 하나라도 reject되면 그 시점에 reject됩니다. “전부 성공해야 할 때”나 “여러 결과를 한 번에 모아 쓸 때” 적합합니다. 하나만 먼저 결과가 나오면 되는 경우에는 **Promise.race**, “전부 끝날 때까지 기다리되 성공/실패 각각 따로”는 **Promise.allSettled**를 사용할 수 있습니다. ES2026에는 **Promise.try(callback)** 이 추가되어, 동기·비동기 함수를 구분 없이 호출해 항상 Promise로 감싸 반환하며, try/catch 보일러플레이트를 줄일 수 있습니다.

</details>

### Q34. async 함수가 Promise 반환 이유
- 의도: 설계 이해
- 키워드: 일관성

<details>
<summary>답변</summary>

async 함수는 **항상 Promise를 반환**합니다. 반환문에 보통 값을 쓰면 그 값으로 **이행된** Promise가 반환되고, 예외가 나면 **거부된** Promise가 반환됩니다. 이렇게 하면 async 함수를 기존 Promise 기반 코드와 동일하게 `.then()`/`.catch()`로 다룰 수 있어 **일관성**이 생깁니다. async 함수끼리 await로 연결할 수도 있고, Promise를 받는 API에 그대로 넘길 수도 있습니다. “비동기 함수 = Promise 반환”이라는 하나의 규칙으로 통일된 설계입니다.

</details>

### Q35. finally 특징
- 의도: finally 이해
- 키워드: 결과 없음, 항상 실행

<details>
<summary>답변</summary>

`.finally(callback)`은 Promise가 **fulfilled든 rejected든** 상관없이 **항상** 한 번 실행됩니다. 인자 없이 호출되고, 이전 Promise의 결과값이나 에러를 그대로 다음 체인으로 넘깁니다. 정리 로직(로딩 끄기, 연결 해제 등)을 성공/실패 공통으로 넣기 좋습니다. 반환값이 있으면 그게 다음 then의 입력이 되고, 없으면 원래 결과가 유지됩니다. try/catch/finally의 finally와 같은 “무조건 실행” 의미입니다.

</details>

## 프로토타입·클래스

### Q36. 객체 리터럴 vs 생성자 함수
- 의도: 객체 생성 방식 이해
- 키워드: 재사용성

<details>
<summary>답변</summary>

**객체 리터럴** `{}`은 한 번 쓰는 객체를 만들 때 적합하고, **생성자 함수**는 같은 형태의 객체를 여러 개 만들 때 씁니다. 생성자 함수는 `new`와 함께 호출해 인스턴스를 만들고, 프로토타입으로 메서드를 공유해 **재사용성**과 메모리 효율을 얻을 수 있습니다. 요즘에는 class 문법으로 생성자+프로토타입을 함께 표현하는 경우가 많고, 단순 데이터 묶음은 리터럴이나 `Object.create`를 쓰는 편입니다.

</details>

### Q37. 프로토타입이란?
- 의도: 객체 모델 이해
- 키워드: 상속 기반

<details>
<summary>답변</summary>

**프로토타입**은 객체가 속성·메서드를 찾을 때 사용하는 **다른 객체에 대한 참조**입니다. 객체에서 프로퍼티를 조회할 때 해당 객체에 없으면, 이 참조를 따라 **프로토타입 객체**를 찾고, 거기에도 없으면 그 프로토타입의 프로토타입으로 올라가는 식으로 **체인**이 이어집니다. 생성자 함수의 `prototype`에 넣어 둔 메서드는 그 생성자로 만든 모든 인스턴스가 공유하므로, JS의 **상속·공유**는 이 프로토타입 체인으로 동작합니다.

</details>

### Q38. 프로토타입 체인
- 의도: 탐색 과정 이해
- 키워드: 링크 구조

<details>
<summary>답변</summary>

객체의 프로퍼티를 참조할 때, 그 객체에 없으면 **`[[Prototype]]`**(또는 `__proto__`로 노출)이 가리키는 객체를 찾고, 거기에도 없으면 다시 그 객체의 프로토타입으로 올라갑니다. 이 **링크를 따라가는 구조**가 프로토타입 체인입니다. 생성자 함수의 `prototype`이 인스턴스의 `[[Prototype]]`이 되므로, `인스턴스 → 생성자.prototype → Object.prototype → null` 순으로 이어집니다. 찾으면 그 시점에 탐색을 멈추고, 끝까지 없으면 `undefined`입니다.

</details>

### Q39. __proto__ vs prototype
- 의도: 내부 구조 이해
- 키워드: 접근자 vs 속성

<details>
<summary>답변</summary>

**`prototype`** 은 **함수 객체**만 갖는 **속성**입니다. 그 함수를 `new`로 호출했을 때 생성되는 객체의 `[[Prototype]]`이 이 `prototype` 객체를 가리킵니다. **`__proto__`** 는 (대부분의 객체가 갖는) **자신의 프로토타입**, 즉 `[[Prototype]]`을 읽고 쓸 수 있게 하는 **접근자 프로퍼티**입니다. 정리하면, `생성자.prototype`은 “이 생성자로 만든 인스턴스의 프로토타입”, `인스턴스.__proto__`는 “이 인스턴스의 프로토타입”이고, `인스턴스.__proto__ === 생성자.prototype`입니다. 공식 API로는 `Object.getPrototypeOf` / `Object.setPrototypeOf` 사용이 권장됩니다.

</details>

### Q40. 프로토타입 메서드 장점
- 의도: 메모리 최적화
- 키워드: 공유

<details>
<summary>답변</summary>

메서드를 **생성자의 prototype**에 두면, 그 생성자로 만든 **모든 인스턴스가 같은 함수 객체를 참조**합니다. 인스턴스마다 메서드를 복사해 두는 것보다 **한 곳만 두고 공유**하므로 메모리 사용이 줄고, 메서드 수정 시 prototype만 바꾸면 모든 인스턴스에 반영됩니다. 인스턴스별로 달라야 할 것은 인스턴스 프로퍼티로 두고, 동일하게 쓰는 동작은 프로토타입 메서드로 두는 식으로 역할을 나누는 것이 좋습니다.

</details>

### Q41. class에서도 프로토타입 알아야 하는 이유
- 의도: 내부 구조 이해
- 키워드: 문법 설탕

<details>
<summary>답변</summary>

class는 **문법 설탕**일 뿐, 내부적으로는 여전히 **생성자 함수 + 프로토타입**입니다. 클래스 메서드는 생성자의 `prototype`에 붙고, `extends`는 프로토타입 체인과 생성자 체인을 연결합니다. 디버깅할 때 `instanceof`, `Object.getPrototypeOf`, 프로퍼티 탐색 순서를 이해하려면 프로토타입이 어떻게 연결돼 있는지 알아야 하고, 레거시 코드나 라이브러리와 연동할 때도 프로토타입을 다루는 경우가 있어서, class만 쓰더라도 프로토타입 개념은 필요합니다.

</details>

### Q42. new 없이 호출 문제
- 의도: 생성자 이해
- 키워드: this 바인딩 깨짐

<details>
<summary>답변</summary>

생성자 함수를 **new 없이** 호출하면 **일반 함수**처럼 동작합니다. 이때 `this`는 (strict가 아니면) 전역 객체나 `undefined`로 바인딩되므로, 생성자 본문에서 `this.xxx`에 할당해도 **새 객체가 아니라 전역이 오염**되거나 에러가 납니다. 또한 **반환값**이 없어 인스턴스가 만들어지지 않습니다. 그래서 생성자 함수는 반드시 `new`와 함께 쓰도록 설계하거나, class를 써서 `new` 없이 호출하면 에러가 나게 하는 것이 안전합니다.

</details>

### Q43. extends 장점
- 의도: 상속 이해
- 키워드: 가독성

<details>
<summary>답변</summary>

**extends**로 상속 관계를 선언하면, 프로토타입 체인과 생성자 체인이 자동으로 잡혀 **“Sub는 Super를 상속한다”**가 코드만 봐도 분명해집니다. 수동으로 `Child.prototype = Object.create(Parent.prototype)` 같은 코드를 쓰는 것보다 **가독성**이 좋고, `super`로 부모 메서드·생성자를 호출하는 것도 문법으로 지원됩니다. 상속 구조를 일관되게 유지하기 쉽고, 정적 메서드 상속도 함께 처리됩니다.

</details>

### Q44. Object.prototype의 상위는?
- 의도: 체인 끝 이해
- 키워드: null

<details>
<summary>답변</summary>

**Object.prototype의 `[[Prototype]]`은 `null`**입니다. 프로토타입 체인은 여기서 끝나며, `Object.prototype`에서도 프로퍼티를 못 찾으면 `undefined`가 됩니다. 모든 일반 객체는 결국 이 체인을 따라 Object.prototype을 거치므로, `toString`, `hasOwnProperty` 같은 메서드를 공통으로 쓰게 됩니다. 체인이 무한히 이어지지 않도록 루트가 `null`로 고정돼 있습니다.

</details>

### Q45. super 이전 this 사용
- 의도: 클래스 초기화 이해
- 키워드: 에러 발생

<details>
<summary>답변</summary>

서브클래스 생성자에서는 **`super()`를 호출하기 전에는 `this`를 사용할 수 없습니다.** 서브클래스 인스턴스는 부모 생성자가 실행된 뒤에 만들어지기 때문에, `super()` 전에는 `this`가 아직 초기화되지 않은 상태입니다. 이때 `this`에 접근하면 **ReferenceError**가 납니다. 부모의 초기화가 끝난 뒤에만 `this`에 프로퍼티를 넣거나 접근할 수 있도록 스펙으로 제한한 것입니다.

</details>

## 배열·문법

### Q46. map/filter/reduce 차이
- 의도: 배열 메서드 이해
- 키워드: 변환, 필터링, 축약

<details>
<summary>답변</summary>

**map**은 각 요소에 콜백을 적용한 **결과로 새 배열**을 만듭니다. 길이는 원본과 같고, 값만 변환(매핑)할 때 씁니다. **filter**는 콜백이 true를 반환하는 요소만 남긴 **새 배열**을 만듭니다. 조건에 맞는 것만 걸러낼 때 씁니다. **reduce**는 배열을 **하나의 값(또는 객체·배열)으로 축약**합니다. 누적자(accumulator)와 현재 요소로 콜백을 호출하고, 반환값이 다음 누적자가 됩니다. 합계, 그룹핑, 플랫닝 등에 쓰입니다. ES2023 이후에는 **비파괴 배열 메서드** `toReversed()`, `toSorted()`, `toSpliced()`, `with(index, value)`가 있어, 원본을 바꾸지 않고 새 배열을 반환하므로 불변성·React 상태 업데이트 등에 활용할 수 있습니다(ES2026 스펙에 포함).

</details>

### Q47. 구조 분해 할당
- 의도: 문법 이해
- 키워드: 패턴 매칭

<details>
<summary>답변</summary>

**구조 분해 할당**은 배열이나 객체의 **패턴에 맞춰** 변수에 값을 풀어 쓸 수 있는 문법입니다. 배열은 `const [a, b] = arr`처럼 순서대로, 객체는 `const { x, y } = obj`처럼 프로퍼티 이름으로 대응합니다. 기본값(`= default`), 나머지(`...rest`), 별칭(`{ name: n }`)도 쓸 수 있어서, 필요한 값만 골라 쓰거나 swap·함수 인자에서 많이 사용합니다.

</details>

### Q48. 스프레드 vs 할당
- 의도: 복사 이해
- 키워드: 얕은 복사

<details>
<summary>답변</summary>

**할당**(`a = b`)은 **참조만 복사**합니다. 객체/배열이면 같은 메모리를 가리키므로, 한쪽을 바꾸면 다른 쪽도 바뀝니다. **스프레드**(`{ ...obj }`, `[...arr]`)는 **새 객체/배열을 만들고** 최상위 프로퍼티·요소만 복사하는 **얕은 복사**입니다. 중첩된 객체는 여전히 참조가 복사되므로, 깊은 복사가 필요하면 재귀·structuredClone 등이 필요합니다. 불변성을 지키려면 스프레드로 새 참조를 만드는 방식이 자주 쓰입니다.

</details>

### Q49. Rest 파라미터
- 의도: 인자 처리 이해
- 키워드: 배열 수집

<details>
<summary>답변</summary>

**Rest 파라미터**는 함수 매개변수에서 `...변수명` 형태로, **나머지 인자들을 하나의 배열**로 받습니다. `function fn(a, b, ...rest)`면 `rest`는 세 번째 인자부터의 배열이 됩니다. arguments 객체 대신 진짜 배열이라 `map`, `length` 등을 바로 쓸 수 있고, 화살표 함수처럼 arguments가 없는 경우에도 사용할 수 있습니다. 인자 개수가 가변일 때 유용합니다.

</details>

### Q50. reduce 초기값 중요성
- 의도: 안정성 이해
- 키워드: 타입 안전

<details>
<summary>답변</summary>

**초기값을 주면** 첫 호출에서 accumulator가 그 값이고, 현재 요소는 배열의 첫 번째 요소부터 사용됩니다. **초기값을 생략하면** accumulator는 배열의 첫 요소, current는 두 번째부터 시작합니다. 빈 배열이면 초기값이 없을 때 **에러**가 나고, 초기값이 있으면 그대로 반환됩니다. 반환 타입을 초기값과 맞추면 **타입 안전**해지고, 빈 배열·단일 요소 배열에서도 동작이 예측 가능해져서, 실무에서는 초기값을 명시하는 편이 안전합니다.

</details>

## ES 모듈·번들·실행

### Q51. ES 모듈 필요성
- 의도: 모듈화 이해
- 키워드: 의존성 관리

<details>
<summary>답변</summary>

ES 모듈은 **파일 단위 스코프**와 **명시적 import/export**로 **의존성 관리**를 표준화합니다. 스크립트 태그만 나열하던 시절에는 전역이 오염되고 로드 순서에 의존했는데, 모듈은 어떤 것이 필요한지·노출하는지가 코드에 드러나고, 번들러·런타임이 의존 관계를 해석해 로드 순서를 보장합니다. 재사용·테스트·유지보수가 쉬워지고, tree-shaking으로 미사용 코드를 제거할 수 있어 번들 최적화에도 유리합니다.

</details>

### Q52. 파일 스코프 장점
- 의도: 격리 이해
- 키워드: 충돌 방지

<details>
<summary>답변</summary>

ES 모듈에서 **최상위 변수·함수는 해당 파일 안에만** 있습니다. 전역 객체에 붙지 않아 **파일 간 이름 충돌**이 줄고, export하지 않으면 다른 파일에서 접근할 수 없어 **격리**가 됩니다. 전역 스코프 오염 없이 내부 구현을 숨길 수 있고, 같은 이름을 다른 모듈에서 안전하게 쓸 수 있어 대규모 프로젝트에서 유지보수가 쉬워집니다.

</details>

### Q53. import alias
- 의도: 이름 충돌 해결
- 키워드: 가독성

<details>
<summary>답변</summary>

**`import { name as alias }`** 또는 **`import * as alias from '...'`** 로 가져온 이름을 **다른 이름으로 쓸 수** 있습니다. 여러 모듈에서 같은 이름을 export할 때 한쪽을 바꿔서 **이름 충돌**을 피하고, 로컬에서 더 의미 있는 이름으로 써서 **가독성**을 높일 수 있습니다. default export도 `import Def as Alias` 형태로 별칭을 줄 수 있습니다.

</details>

### Q54. default vs named export
- 의도: 모듈 설계
- 키워드: 유연성

<details>
<summary>답변</summary>

**default export**는 모듈당 하나만 두고, `import Foo from '...'`처럼 **이름을 자유롭게** 정할 수 있습니다. 주로 그 모듈의 “메인” 하나(컴포넌트, 클래스 등)를 노출할 때 씁니다. **named export**는 여러 개를 `export { a, b }`로 내보내고, `import { a, b }`로 **이름을 맞춰** 가져옵니다. 필요한 것만 골라 가져올 수 있어 tree-shaking에 유리하고, default와 named를 섞어 쓸 수 있어 **유연성**이 있습니다. 한 모듈에서 한 가지를 메인으로 두고 나머지는 named로 두는 패턴이 많습니다.

</details>

### Q55. dynamic import
- 의도: 지연 로딩
- 키워드: 코드 분할

<details>
<summary>답변</summary>

**`import()`** 는 **함수 형태**로 모듈을 **동적으로** 불러옵니다. 호출 시점에 네트워크에서 해당 모듈을 가져오고, **Promise**를 반환하므로 `then`이나 async/await로 사용합니다. 초기 로딩 시에는 불러오지 않고, 라우트 진입이나 조건 충족 시에만 로드하는 **지연 로딩**에 쓰이고, 번들러가 이 호출을 기준으로 **코드 분할**해 별 청크를 만들 수 있어 초기 번들 크기를 줄일 수 있습니다.

</details>

### Q56. dynamic import 성능
- 의도: 최적화 이해
- 키워드: 초기 로딩 감소

<details>
<summary>답변</summary>

dynamic import로 진입 경로별·기능별로 청크를 나누면 **초기 로딩 시 받는 JS 양이 줄어듭니다.** 사용자가 당장 쓰지 않는 코드는 나중에 필요할 때만 불러오므로 FCP·TTI가 좋아질 수 있습니다. 대신 해당 경로 첫 방문 시 추가 요청이 발생해 지연이 생길 수 있으므로, 핵심 경로는 미리 로드하고 부가 기능만 지연 로드하는 식으로 나누는 것이 일반적입니다.

</details>

### Q57. 번들링 필요성
- 의도: 배포 이해
- 키워드: 파일 최적화

<details>
<summary>답변</summary>

개발 시에는 모듈 단위로 나누지만, 브라우저에 **요청 수가 많으면** 로딩·파싱 비용이 커집니다. **번들링**은 여러 모듈을 **의존 관계에 맞춰 하나 또는 소수의 파일**로 합쳐, HTTP 요청 수를 줄이고 **파일 최적화**(압축, tree-shaking, 코드 분할)를 적용하기 위함입니다. 번들러(Webpack, Vite 등)가 import/export를 해석해 하나의 실행 가능한 그래프로 만들고, 프로덕션에서는 이렇게 만든 청크를 배포합니다.

</details>

### Q58. JS 실행 단계
- 의도: 엔진 이해
- 키워드: 파싱, 컴파일, 실행

<details>
<summary>답변</summary>

대략 **파싱 → 컴파일(또는 인터프리트) → 실행** 단계로 나뉩니다. **파싱**에서 소스 코드를 **AST(추상 구문 트리)** 로 만듭니다. 그다음 **컴파일** 단계에서(최신 엔진은 JIT) 바이트코드나 기계어로 변환하고, **실행** 단계에서 실행 컨텍스트를 만들고 바이트코드를 실행합니다. V8 등은 파싱·컴파일을 점진적으로 하고, 자주 쓰는 코드를 최적화하는 방식으로 동작합니다.

</details>

## DOM·이벤트

### Q59. DOM이란?
- 의도: 브라우저 모델 이해
- 키워드: 트리 구조

<details>
<summary>답변</summary>

**DOM(Document Object Model)** 은 HTML(또는 XML) 문서를 **노드들의 트리 구조**로 표현한 것입니다. 문서 객체라 부르며, JavaScript로 이 트리의 노드를 조회·수정·추가·삭제해 화면과 상호작용합니다. 루트는 `document`, 각 태그는 요소 노드, 텍스트는 텍스트 노드로 표현되고, 부모-자식 관계로 계층이 이어집니다. 브라우저는 이 모델을 렌더 트리와 연결해 레이아웃·페인트를 하고, DOM 변경 시 리플로우·리페인트가 일어날 수 있습니다.

</details>

### Q60. 이벤트 전파 3단계
- 의도: 이벤트 흐름 이해
- 키워드: 캡처, 타겟, 버블

<details>
<summary>답변</summary>

이벤트는 **캡처 → 타겟 → 버블** 세 단계로 전파됩니다. **캡처 단계**에서는 document부터 **타겟까지 위에서 아래로** 내려가며, **타겟 단계**에서는 이벤트가 발생한 요소에서 처리되고, **버블 단계**에서는 타겟부터 **document까지 아래에서 위로** 올라갑니다. `addEventListener`는 세 번째 인자로 `true`를 주면 캡처에서, 기본값 `false`면 버블에서 리스너가 호출됩니다. 대부분 버블을 사용하고, 상위에서 위임 처리할 때 활용합니다.

</details>

### Q61. async vs defer
- 의도: 로딩 전략 이해
- 키워드: 실행 타이밍

<details>
<summary>답변</summary>

**async** 스크립트는 **다운로드가 끝나면 HTML 파싱을 멈추고 즉시 실행**됩니다. 순서는 보장되지 않습니다. **defer** 스크립트는 **다운로드 중에는 파싱을 계속**하고, **DOMContentLoaded 직전에 문서 순서대로 실행**됩니다. 둘 다 스크립트가 파싱을 블로킹하지 않는다는 점은 같고, 의존 관계가 있거나 DOM이 준비된 뒤 실행해야 하면 defer, 독립적인 스크립트면 async를 쓰는 식으로 구분합니다.

</details>

### Q62. preventDefault vs stopPropagation
- 의도: 이벤트 제어
- 키워드: 기본 동작, 전파 중단

<details>
<summary>답변</summary>

**preventDefault()** 는 해당 이벤트의 **기본 동작**을 막습니다. 예를 들어 링크 클릭 시 이동, 폼 제출 시 새로고침을 막을 때 씁니다. **stopPropagation()** 은 **캡처/버블 전파를 중단**해서, 부모나 자식 노드의 리스너가 그 이벤트를 받지 못하게 합니다. 같은 노드의 다른 리스너는 그대로 호출됩니다. 기본 동작만 막으려면 preventDefault, 상위로 이벤트가 올라가는 것만 막으려면 stopPropagation을 쓰면 됩니다.

</details>

### Q63. DOM 성능 최적화
- 의도: 렌더링 이해
- 키워드: 리플로우 최소화

<details>
<summary>답변</summary>

DOM 변경은 **레이아웃(리플로우)·리페인트**를 유발할 수 있어, 자주 바꾸면 성능이 떨어집니다. **리플로우를 최소화**하려면: 여러 스타일·구조 변경을 한 번에 묶고, 레이아웃에 영향을 주는 속성(width, height, position 등) 읽기를 배치 앞쪽에서 하고 쓰기는 나중에 모아서 하며, 가능하면 `transform`·`opacity`처럼 컴포지트만 쓰는 변경으로 처리합니다. 또 DocumentFragment로 여러 노드를 모아 한 번에 붙이고, 가상 스크롤로 보이는 구간만 DOM에 두는 방식도 자주 씁니다.

</details>

### Q64. 이벤트 버블링 제어
- 의도: 이벤트 흐름 제어
- 키워드: stopPropagation

<details>
<summary>답변</summary>

자식에서 발생한 이벤트가 **부모로 올라가는 것(버블링)** 을 막으려면, 해당 리스너 안에서 **`event.stopPropagation()`** 을 호출합니다. 그러면 그 이벤트는 더 이상 상위로 전파되지 않습니다. 특정 영역 안의 클릭만 처리하고 상위 리스너(예: 문서 클릭으로 닫는 모달)가 반응하지 않게 할 때 사용합니다. 전파만 막는 것이므로 기본 동작(링크 이동 등)은 **preventDefault**로 따로 막아야 합니다.

</details>

### Q65. 번들 파일 로딩 전략
- 의도: 성능 최적화
- 키워드: 분할 로딩

<details>
<summary>답변</summary>

**분할 로딩**으로 초기 번들을 줄이는 전략을 씁니다. **코드 스플리팅**: 라우트·기능별로 dynamic import해 청크를 나누고, 해당 경로 진입 시에만 로드합니다. **벤더 청크 분리**: React 등 큰 라이브러리는 별 청크로 빼서 캐시를 오래 쓰게 합니다. **프리로드/프리페치**: 중요 리소스는 `<link rel="preload">`, 다음에 쓸 가능성이 있으면 `prefetch`로 미리 가져옵니다. 이렇게 해서 FCP·TTI를 줄이고 캐시 효율을 높입니다.

</details>

### Q66. 캡처링 단계 처리 방법
- 의도: 이벤트 단계 활용
- 키워드: addEventListener 옵션

<details>
<summary>답변</summary>

**캡처링 단계**에서 리스너를 실행하려면 `addEventListener`의 **세 번째 인자에 `true`** 를 넘깁니다. `element.addEventListener('click', handler, true)`처럼 쓰면, 해당 이벤트가 document에서 타겟으로 내려가는 **캡처 단계**에서 handler가 호출됩니다. 기본값은 `false`(버블 단계)입니다. 자식보다 먼저 상위에서 처리해야 할 때(예: 포커스 트랩, 전역 로깅) 캡처를 사용합니다. 옵션 객체 `{ capture: true }`로도 지정할 수 있습니다.
